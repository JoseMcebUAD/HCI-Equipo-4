## TraceabilityAtt

Nowadays, building software isn't just about making sure the code runs; it’s about actually stepping into the user's shoes. It doesn't matter how many features you have if the person using the app feels lost or keeps hitting walls. That’s why, in our prototype, we went all-in on Error Prevention. The idea is simple: instead of letting the user make a mistake and then hitting them with a "hey, you messed up" error message, we designed the interface so they can’t trip up in the first place. For the scheduling part, we automatically gray out dates that have already passed or time slots that are already taken. This way, the system guides you down the right path and saves you the headache of dealing with annoying red pop-ups.

To figure out if this was actually working, we didn't just go with our "gut feeling." We used a metric called the Rate of Avoidable Errors (M-01). Basically, we watched people use the app and counted how many times they tried to do something they weren't supposed to, like clicking a button that was supposed to be off-limits. We chose this data because it gives us the cold, hard truth: if a lot of people are trying to click something that's disabled, it means our design isn't doing a good job of showing that the option is closed. It’s a reality check to see if the app is actually intuitive or if it only makes sense to the developers.

When it came to gathering this info, we did it by taking notes while watching users, but there are other ways to handle it too. We could have used analytics tools to automatically track every time someone tries to interact with a disabled element, or measured how many people finished the whole process "clean"—without a single warning popping up. In the end, our goal is to keep those errors under 5%. We want using the app to feel so smooth that it’s like the system is looking out for the user, stopping problems before they even happen.
