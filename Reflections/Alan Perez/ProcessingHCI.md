# Lean UX Technique

There are a couple of key points of the Lean UX technique: Focus on creating changes in user behavior, not just deliver screens; Every role is working together since the beginning to get a better understanding in the project; Every new idea is a guess until it gets validated by real users; Build the smallest thing (MVP) first to test assumptions quick; Engage with users throughout the entire development process; Launch early, fail fast, and then adapt quickly.

As a team we didn’t follow the key points of Lean UX, and now that I acknowledge, it could be helpful for our development process. We didn’t focus on creating outcomes, just finishing the artifacts that were asked for the first delivery; Sometimes we worked together on ideas, but most of the time each member had a role and a personal delivery to attend; For now we haven’t worked on an MVP, but probably we wouldn’t try to make a quick version for the prototype, we would share the work and divide tasks for each member to make a rather complete version; We haven’t made quick iterations, just do it once and call it done without much feedback.

To be considered Lean, as a team we would have to make some changes: Stop working only in isolated roles, everyone should be involved in the whole process; Stop waiting for something to be complete to ask for feedback, have regular user testing, get our work in front of people even if it’s unfinished; Changing the workflow to be daily, not getting tasks that would take several days or even weeks. Adopting Lean UX requires a lot of communication, and it could be hard to find users to test our project continuously, but there are some benefits, the biggest one is risk reduction, investing a lot of time making something the best possible only to discover that users don’t consider it easy to use could be detrimental for the project progress. <mark>In Lean, spending weeks polishing a complete version without real-world validation is considered a waste, because it doesn't result in actual learning.</mark>
